## R3: Modern Reactive Framework for Godot C#
R3 (New generation of UniRx) is the backbone for state management and UI binding. It transforms "Data Changes" and "User Inputs" into predictable, auto-disposable **Data Streams**.
### 1. Reactive Properties (State Management)
Don't just use events; use `ReactiveProperty<T>` to wrap your data.
- **Automatic Initialization**: UI subscribes and immediately gets the current value (no more manual `UpdateUI()` on start).
- **Distinct Changes**: Only triggers when the value actually changes.
```csharp
// In your Logic/Component:
public readonly ReactiveProperty<int> Health = new(100);
// In your UI (UIBase):
Health.Subscribe(hp => healthBar.Value = hp).AddTo(DisposeBag);
2. Godot Native Extensions (Convenience)
Use R3.Godot built-in extension methods instead of verbose FromEvent.
C#

// Buttons
_closeButton.PressedAsObservable()
    .Subscribe(_ => HideUI())
    .AddTo(DisposeBag);
// Sliders / LineEdit
_volumeSlider.ValueChangedAsObservable()
    .Subscribe(v => _audioService.SetVolume(v))
    .AddTo(DisposeBag);
// Frame-based Logic (Uses Godot's _Process)
Observable.EveryUpdate()
    .Subscribe(_ => _label.Text = $"Time: {Time.GetTicksMsec()}")
    .AddTo(DisposeBag);
3. Advanced Stream Manipulation (The Power of Rx)
This is where R3 shines over traditional C# events:
C#

// Debounce: Prevents button spamming (wait 0.5s after last click)
_saveButton.PressedAsObservable()
    .ThrottleFirst(TimeSpan.FromSeconds(0.5)) 
    .Subscribe(_ => SaveGame())
    .AddTo(DisposeBag);
// CombineLatest: Complex UI conditions (e.g., Enable button only if Form is valid)
Observable.CombineLatest(
    _nameInput.TextChangedAsObservable(),
    _ageInput.ValueChangedAsObservable(),
    (name, age) => !string.IsNullOrEmpty(name) && age >= 18
)
.Subscribe(isValid => _submitButton.Disabled = !isValid)
.AddTo(DisposeBag);
4. Asynchronous & Threading
Always ensure UI updates happen on the Godot Main Thread.
C#

// Fetch data from Web/Thread and jump back to Godot Main Thread
Observable.FromAsync(async ct => await MyAsyncApi.GetData(ct))
    .ObserveOn(GodotProvider.MainThread) // CRITICAL for Godot UI safety
    .Subscribe(data => _label.Text = data)
    .AddTo(DisposeBag);
5. Architectural Rules (Mandatory)
	1. The "DisposeBag" Rule:
		○ Every UI class must have a CompositeDisposable DisposeBag.
		○ Every .Subscribe() must end with .AddTo(DisposeBag).
		○ DisposeBag.Dispose() must be called in _ExitTree().
	2. The "Clean Component" Rule:
		○ High-frequency physics data (Velocity, Position) should remain as Standard C# Properties for performance.
		○ Discrete events (OnDeath, OnItemCollected) and UI states (IsMenuOpen, CurrentHealth) should be ReactiveProperties.
	3. Validation & Filtering:
		○ Use .Where(x => ...) to filter events BEFORE they hit the UI.
		○ Use .DistinctUntilChanged() to avoid redundant UI redraws.
6. Installation Check
Ensure your .csproj includes:
	• R3
	• R3.Godot (Provides GodotProvider and Frame-based operators)

### 补全说明（你可以告诉 Claude 的点）：

* **PressedAsObservable() 等扩展**：这极大地简化了代码，不需要手写那三行 `FromEvent`。
  * **ObserveOn(GodotProvider.MainThread)**：这在做异步网络请求或多线程加载（比如加载背包 Excel 数据）时是**救命**的，防止 Godot 崩溃。
  * **CombineLatest**：这是处理“背包结算”等复杂逻辑的神技，能把多个数据流合并成一个判断结果。
  * **ThrottleFirst**：这是处理按钮防抖（防止玩家点太快）的最优雅方式。

R3 高级技术细节与优化指南
1. 极其关键的 API (Claude 易遗漏项)
	• AsUnitObservable():
		○ 用途: 当你只关心事件“发生了”，而不关心事件带的参数（比如把 float 变成 Unit）时使用。
		○ 场景: 滑块数值改变触发保存动作，但保存函数不需要滑块的值。
	• DistinctUntilChanged():
		○ 性能优化: 核心 API。只有当新值与旧值不同时，才向下游派发。
		○ 场景: IsGrounded 每一帧都在赋值，但只有在“落地”或“起跳”那一瞬间，UI 才需要响应。
	• Chunk(TimeSpan) / Buffer(int count):
		○ 用途: 将高频事件打包。
		○ 场景: 0.5秒内受到的多次伤害数值合并成一个总数显示。
	• CombineLatest:
		○ 用途: 组合多个流的最新值。
		○ 场景: 背包结算。当（物品A位置改变 OR 物品B位置改变）时，重新计算两者的连携效果。
	• Select vs SelectMany:
		○ 细节: Select 用于转换数值，SelectMany 用于异步平铺（比如点击按钮后发起一个异步网络请求）。
2. 内存与生命周期管理 (The "Leak-Proof" Protocol)
	• CompositeDisposable: 必须作为 UI 类的成员变量，它是订阅信号的“收纳盒”。
	• AddTo(DisposeBag):
		○ 细节: 这是 R3.Godot 的扩展。它能自动关联到 Godot 节点的 tree_exiting 信号。
		○ 规则: 除非订阅是全局单例且永不销毁，否则禁止出现不带 .AddTo() 的 .Subscribe()。
	• Dispose() 幂等性: DisposeBag.Dispose() 可以在 _ExitTree() 中放心调用多次，它是安全的。
3. 性能优化与“洋葱架构”准则
	• 拉取 (Pull) 优于 推送 (Push):
		○ 原则: 每一帧都在变的物理数据（如 Velocity）不要做成 ReactiveProperty。
		○ 正确做法: UI 使用 Observable.EveryUpdate().Select(_ => moveComp.Velocity) 来主动观察。这避免了底层组件每一帧都在 setter 里做相等检查和订阅者遍历。
	• 分配优化 (Zero-GC):
		○ 尽量在 _Ready 中定义好流的链式结构并存储起来，避免在 _Process 或循环中创建新的 Observable 实例。
4. 异步处理与线程安全 (Async Integration)
	• ObserveOn(GodotProvider.MainThread):
		○ 极其关键: 当你从 Task.Run 或网络库返回数据时，必须调用此 API 才能操作 Godot UI 控件，否则会引发多线程崩溃。
	• CancellationToken:
		○ R3 完美支持取消令牌。在 UI 销毁时，所有关联的异步流会自动 Cancel。
5. 常用组合操作符速查表
操作符	描述	最佳 UI 场景
ThrottleFirst	触发一次后屏蔽一段时间	防止按钮连点、疯狂开门
Debounce	停止操作后一段时间触发	输入搜索框，打完字才搜索
Pairwise	获取（上一次值，当前值）	计算血量扣除的差值动画
Where	过滤流	只有血量 < 20% 时才让屏幕闪红光
Merge	合并多个同类型流	多个按钮点击都触发同一个关闭界面动作

指导 Claude 重构的指令示例：
当你让 Claude 改进代码时，可以使用以下指令加强针对性：
	"请使用 R3 优化这段 UI 代码。特别注意使用 DistinctUntilChanged() 避免不必要的 UI 刷新。如果需要组合多个数据状态，请使用 CombineLatest。对于按钮点击，请加入 ThrottleFirst 防抖处理。所有订阅必须加入 DisposeBag 并在 _ExitTree 中销毁。"
为什么这对你的项目很重要？
	1. 复用性：如果你在做“背包乱斗”类型的系统，CombineLatest 和 DistinctUntilChanged 是实现物品之间“连锁检测”的唯一优雅方式。
	2. 技术细节：Claude 往往只给你写最简单的 Subscribe，如果不加 DistinctUntilChanged，你的进度条可能会在数值没变的情况下每一帧都重新赋值，这在 Godot 里会造成不必要的绘图重绘开销。
	3. 便捷性：使用 R3.Godot 提供的原生扩展（如 PressedAsObservable）可以让你的代码行数减少 40% 以上。
