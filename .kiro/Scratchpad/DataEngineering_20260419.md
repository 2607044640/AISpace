# 数据工程优化 - AI 上下文合并脚本

## 优化日期
2026-04-19

## 问题诊断

### 发现的问题
1. **低价值文件污染**：LICENSE.txt、.sln、.editorconfig 等文件对 AI 理解代码架构无帮助
2. **文件大小失控**：大型 .tscn 文件（>200KB）包含大量节点序列化数据，消耗 token 但无实际价值
3. **编码崩溃风险**：未强制 UTF-8 编码，遇到多语言注释会导致 UnicodeDecodeError
4. **空文件浪费**：空的 .cs 或 .md 文件浪费分隔符空间
5. **统计信息不足**：无法了解每个 section 的文件数和行数分布

### 根本原因
- 脚本设计时未考虑 RAG 系统的 token 成本和注意力机制
- 缺少针对不同文件类型的差异化处理策略
- 缺少鲁棒的编码处理

## 架构决策

### 决策 1：分级文件体积阈值策略

**理由**：
- 纯代码文件（.cs, .gd）：超过 1MB 通常是机器生成或包含大量重复逻辑，阅读价值递减
- Godot 资源文件（.tscn, .tres, .json）：超过 200KB 几乎全是节点序列化数据，对理解架构无帮助
- 配置文件（.cfg, .xml）：通常很小，不需要特殊限制

**实施**：
```python
CODE_MAX_SIZE_MB = 1.0      # 纯代码最大 1MB
RESOURCE_MAX_SIZE_MB = 0.2  # 场景/资源最大 200KB
```

### 决策 2：文件类型黑名单

**排除的文件类型**：
- `.sln` - Visual Studio Solution 文件，仅 IDE 配置
- `.editorconfig` - 编辑器格式化规则
- `.user` - 用户特定配置
- `.DotSettings.user` - Rider 用户配置

**排除的特定文件名**：
- `LICENSE.txt` - 法律文本，对代码理解无用
- `test_paths.txt` - 临时测试文件
- `debug.txt` - 调试日志
- `temp.txt` - 临时文件

### 决策 3：强制 UTF-8 编码

**理由**：
- Godot 项目默认使用 UTF-8 编码
- Windows 系统 Python 默认可能使用 GBK
- 项目中存在中文、日文注释

**实施**：
```python
with open(file_path, 'r', encoding='utf-8', errors='ignore') as f:
```

### 决策 4：空文件过滤

**理由**：
- 空文件浪费分隔符行（4行 `====` 标记）
- 对 AI 理解无任何贡献

**实施**：
```python
if not content.strip():
    skipped_files_count += 1
    continue
```

### 决策 5：分段统计信息

**理由**：
- 帮助用户了解每个 section 的内容分布
- 便于调试和优化过滤规则

**实施**：
- 分别统计 Section 1（规则文档）和 Section 2（项目代码）的文件数和行数
- 在最终输出中显示详细统计

## 预期效果

### Token 节省
- 排除 LICENSE 和 IDE 配置文件：预计节省 5-10%
- 大文件阈值控制：预计节省 20-30%
- 空文件过滤：预计节省 2-5%
- **总计预计节省 30-45% 的无效 token**

### 质量提升
- 更高的信噪比
- 更精准的代码上下文
- 减少 AI "幻觉"风险
- 提升 RAG 检索准确度

### 鲁棒性提升
- 消除编码崩溃风险
- 更可靠的自动化流程

## 技术债务

### 未来可能的优化
1. **智能内容分析**：基于文件内容而非仅大小判断价值
2. **增量更新**：只合并变更的文件，而非每次全量
3. **压缩策略**：对重复内容（如 using 语句）进行去重
4. **语义分块**：将超大文件拆分为有意义的代码块

### 监控指标
- 合并文件总大小
- 每个 section 的文件数和行数
- 被过滤文件的统计（按原因分类）
- 生成时间
